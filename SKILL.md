# Idea to Build

アイデアベースの依頼を受けたら、いきなりコードを書かず、軽量ヒアリング → スペック整理 → 技術計画 → 実装 の順で進める。

## Phase 1: Quick Hearing

依頼内容から不足している情報を特定し、AskUserQuestion で補う。
**すでに明確な項目は聞かない。** 必要な質問だけを選択する（最大4問/回、1-2回で完了）。

### 確認すべき4項目

1. **What**: 何を作るか（具体的な機能・振る舞い）
2. **Who**: 誰が使うか（自分用 / チーム / エンドユーザー）
3. **Why**: 何を解決するか（ペインポイント・動機）
4. **How**: 技術的な制約・好み（言語、フレームワーク、既存コードの有無）

### ヒアリングのルール

- 依頼文から読み取れる項目はスキップ（例: 「PythonでCLIツール作って」→ How は確定）
- 自分用ツールの場合は Who/Why を深掘りしない
- 「特に決めていない」回答には 💡 デフォルト提案 + 理由を1文で添える
- 最大2回の AskUserQuestion で完了させる（長いヒアリングにしない）

## Phase 2: Spec Crystallization

ヒアリング結果をユーザーストーリー + MVP スコープとして整理し、ユーザーに提示する。

### 出力フォーマット

以下の形式でテキスト出力し、ユーザーの承認を得る：

```
## スペック確認

**ユーザーストーリー**
> As a [ユーザー], I want [機能], so that [価値]

**MVP スコープ (Must Have)**
- [ ] 機能1
- [ ] 機能2

**後回し (Nice to Have)**
- [ ] 機能3

**技術方針**
- 言語/FW: [選定]
- 主要な設計判断: [判断内容]
```

修正要望があれば反映してから次フェーズへ進む。

## Phase 3: Technical Planning

スペック承認後、実装計画を立てる。

### 規模に応じた判断

- **1-2ステップで完了**: 即実装（Plan Mode 不要）
- **3ステップ以上**: EnterPlanMode で計画提示してから実装
- **設計判断あり**: 2-3のアプローチを比較提示し、選択を得る

### 計画に含めること

- ファイル構成・ディレクトリ設計
- 主要な実装ステップ
- 既存コードへの影響範囲（該当する場合）

## Phase 4: Execute

承認された計画に基づいて実装する。

- CLAUDE.md の Quality Gates を適用（セキュリティ・エッジケース・テスト）
- 実装完了後、動作確認を行う
- UI/デザイン要素が大きい場合は `biz-context-hearing` の利用を提案

## スキップ判断

以下の場合はヒアリングを省略し、Phase 2 or 3 から開始してよい：

- 依頼が十分に具体的（What/Who/Why/How すべて明確）
- 既存機能の小さな拡張（コンテキストが自明）
- ユーザーが「すぐ作って」「ヒアリング不要」と明示した場合
